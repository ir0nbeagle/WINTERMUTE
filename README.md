# WINTERMUTE

**Threat Hunting · Threat Intelligence · Detection Engineering**

WINTERMUTE is an open-source, file-based threat hunting and intelligence tracking platform built on GitHub. Hunt packages live as versioned Markdown, metrics are auto-generated by a Python script, and dashboards are served as static HTML via GitHub Pages — no database, no server, no SaaS dependency.

> **Live dashboards →** [ir0nbeagle.github.io/WINTERMUTE](https://ir0nbeagle.github.io/WINTERMUTE/)

---

## Why WINTERMUTE?

Most threat hunting programs live in spreadsheets or proprietary tools. WINTERMUTE treats hunts and intel the same way software teams treat code: version-controlled, peer-reviewable, metric-driven, and portable.

- **Detection as Code** — every hunt is a `LOCK.md` file. Git history is your audit trail.
- **Intel-to-Hunt pipeline** — threat intel reports (`TIR-*.md`) drive hunt creation. Coverage gaps are surfaced automatically.
- **No infrastructure** — runs entirely on GitHub + GitHub Pages. Free forever.
- **Executive-ready** — live dashboards with MITRE coverage, hunt-to-detection rates, APT profiles, and IOC exports.

## Core Concepts

### The LOCK Framework

Every hunt is structured as a `LOCK.md` file with five sections:

| Section | Purpose |
|---------|---------|
| **LEARN** | Threat context, actor profile, MITRE mapping |
| **OBSERVE** | What data sources and log telemetry are needed |
| **CHECK** | The actual hunt queries (Falcon, YARA-L, KQL, etc.) |
| **KEEP** | Findings, outcome, detection rule created |
| **METRICS** | Structured checkboxes parsed by `parse_metrics.py` |

The METRICS section is machine-readable and drives all dashboard data — hunt outcome, detection conversion, visibility gaps, actor attribution.

### Threat Intel Reports (TIR)

Intel reports live in `threat-intel/TIR-YYYY-NNN-<slug>/report.md`. They use YAML frontmatter to carry structured metadata:

```yaml
---
id: TIR-2026-002
title: "LockBit 3.0 Healthcare Targeting"
date: 2026-04-15
analyst: ir0n
tlp: AMBER
priority: High
status: hunt-created        # new | in-review | hunt-created | no-hunt | monitoring
linked_hunts: [H-0030]
threat_actors:
  - name: LockBit 3.0
    type: cybercriminal
techniques: [T1021.002, T1570, T1486]
tools:
  - name: Cobalt Strike
iocs:
  - type: ip
    value: "185.220.101.47"
    context: C2 server observed in healthcare sector campaign
    tlp: AMBER
---
```

The body is free-form Markdown: executive summary, technical analysis, MITRE table, recommendations.

### Detection as Code

Hunt files are the source of truth. The pipeline is:

```
Intel Report (TIR) → Hunt Created (LOCK.md) → Detection Rule → Metric Updated
```

GitHub Actions automatically parse `LOCK.md` files on every push and update `metrics/summary.json`. Dashboards read from that JSON — no build step required.

## Quick Start

### Fork and clone

```bash
git clone https://github.com/<your-org>/WINTERMUTE.git
cd WINTERMUTE
pip install -r requirements.txt
```

### Enable GitHub Pages

In your repo settings: **Pages → Source → Deploy from branch → `main` / `(root)`**

The `.nojekyll` file at the repo root tells GitHub to serve files as-is without Jekyll processing.

### Run the metrics parser

```bash
python3 -B scripts/parse_metrics.py
```

This reads all `LOCK.md` and `report.md` files and writes `metrics/summary.json`. Run it after adding or updating any hunt or intel report. GitHub Actions runs it automatically on push.

### Export IOCs

```bash
python3 -B scripts/export_iocs.py
```

Reads IOCs from all TIR YAML frontmatter and writes `exports/stix-bundle.json`, `exports/misp-export.json`, and `exports/iocs.csv`. Respects TLP — RED IOCs are excluded from the STIX bundle by default.

## Writing a Hunt

### 1. Create the directory

Name it `H-NNNN-<short-slug>` — increment the number from the last hunt.

```bash
mkdir hunts/H-0039-My-Hunt-Name
```

### 2. Copy the template

The easiest way is to copy an existing hunt and modify it. The METRICS section at the bottom drives all dashboard data — keep its structure intact.

```bash
cp hunts/H-0029-TeamPCP-PyPI-AI-ML-Supply-Chain/LOCK.md hunts/H-0039-My-Hunt-Name/LOCK.md
```

### 3. Fill in the sections

**Required fields in the header:**

```markdown
**Hunt ID:** H-0039
**Date Created:** 2026-06-17
**Status:** In Progress          # In Progress | Completed | On Hold
**Analyst:** your-handle
**Source:** <link to intel or internal trigger>
**Platform:** CrowdStrike Falcon / Google SecOps SIEM
**ATT&CK Techniques:** T1059.001 — PowerShell, T1055 — Process Injection
**ATT&CK Tactics:** Execution, Defense Evasion
**Priority:** High               # Critical | High | Medium | Low
```

**METRICS section checkboxes** (the parser reads these):

```markdown
## METRICS

**Threat Actor:**
- [ ] Scattered Spider
- [ ] FIN7
- [x] Other: BlackCat/ALPHV — **BlackCat/ALPHV** (RaaS operator)

**Hunt Outcome:**
- [x] True Positive
- [ ] False Positive
- [ ] No Findings
- [ ] Pending

**Converted to Detection Rule:**
- [x] Yes — Rule Name: BLACKCAT — Suspicious VSS Deletion via cmd.exe
- [ ] No

**Converted to Hunting Rule:**
- [ ] Yes
- [x] No

**Log/Visibility Gap Found:**
- [x] Yes — Gap: EDR not deployed on legacy OT network segment
- [ ] No
```

### 4. Add query files (optional)

```bash
# CrowdStrike Falcon
touch hunts/H-0039-My-Hunt-Name/crowdstrike-queries.md

# Google SecOps / Chronicle YARA-L
touch hunts/H-0039-My-Hunt-Name/google-secops-queries.md
```

### 5. Push — metrics auto-update

```bash
git add hunts/H-0039-My-Hunt-Name/
git commit -m "hunt: H-0039 <short description>"
git push
```

The `parse-metrics` GitHub Action runs automatically, updates `metrics/summary.json`, and the dashboards reflect the new hunt within ~60 seconds.

## Writing an Intel Report

### 1. Copy the template

```bash
cp -r threat-intel/TEMPLATE threat-intel/TIR-2026-010-Actor-Campaign-Name
```

### 2. Fill in the YAML frontmatter

The frontmatter is machine-readable — keep the field names exactly as shown. The `status` field drives the Hunt Queue funnel:

| Status | Meaning |
|--------|---------|
| `new` | Received, not yet reviewed |
| `in-review` | Analyst actively reviewing |
| `hunt-created` | One or more LOCK.md hunts created |
| `no-hunt` | Reviewed, no hunt warranted |
| `monitoring` | Watching for additional indicators |

### 3. TLP guidance

| TLP | Use for |
|-----|---------|
| `WHITE` | Public information — IOCs safe to share |
| `GREEN` | Share within community, not publicly |
| `AMBER` | Share on need-to-know basis (default) |
| `RED` | Not for distribution — excluded from STIX export |

### 4. IOC types supported

`ip`, `domain`, `url`, `sha256`, `md5`, `sha1`, `filename`, `email`, `package`, `regex`, `unicode-range`

## GitHub Actions

| Workflow | Trigger | What it does |
|----------|---------|-------------|
| `parse-metrics.yml` | Push to `hunts/**/LOCK.md` or `threat-intel/**/report.md` | Regenerates `metrics/summary.json`, commits back |
| `export-iocs.yml` | Push to `threat-intel/**/report.md` | Regenerates STIX, MISP, CSV exports, commits back |
| `validate-hunt.yml` | Pull request touching `hunts/` | Validates LOCK.md has required fields and METRICS section |
| `weekly-report.yml` | Cron (Mondays 08:00 UTC) | Prints weekly metrics digest to Actions log |

All workflows use `python3 -B` to avoid `__pycache__` issues on mounted filesystems.

## Platform Support

WINTERMUTE ships with query templates for:

- **CrowdStrike Falcon** — LogScale Query Language (LQL) (`crowdstrike-queries.md`)
- **Google SecOps / Chronicle** — YARA-L 2.0 detection rules (`google-secops-queries.md`)

Adding support for other platforms (Elastic, Sentinel, Splunk ES, QRadar) means adding a `<platform>-queries.md` file to the hunt directory and updating the hunt viewer tabs in `reports/hunt.html`.

## Contributing

Contributions welcome — new hunts, intel reports, platform query templates, dashboard improvements.

**To contribute a hunt:**
1. Fork the repo
2. Create `hunts/H-NNNN-<slug>/LOCK.md` following the format above
3. Add platform query files if available
4. Open a PR — the validate workflow checks structure automatically

**To contribute an intel report:**
1. Fork the repo
2. Create `threat-intel/TIR-YYYY-NNN-<slug>/report.md`
3. Ensure TLP is set appropriately — use `GREEN` or `WHITE` for public contributions
4. Open a PR

**Hunt numbering:** Check the highest `H-NNNN` in `hunts/` and increment. If there's a conflict in a PR, the reviewer will renumber before merging.

## Philosophy

> *"A hunt that isn't written down didn't happen."*

WINTERMUTE is opinionated: every hunt gets a LOCK.md, every intel report drives a hunt or is explicitly closed. The TTP gap metric — techniques identified in intel with no corresponding hunt — is the backlog. Shrinking it is the job.

Detection as Code means hunts are peer-reviewed, tested, and version-controlled the same way production code is. A detection rule without a hunt package is a rule without a story.
